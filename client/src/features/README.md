# `features/` 디렉터리

사용자 액션을 처리하는 **비즈니스 로직과 UI**를 구현하는 레이어입니다.
로그인, 타이머 제어, 인증 등 **하나의 기능 단위**를 완결적으로 담당합니다.

---

## 1. 역할 · 책임

| 구분              | 설명                                      |
| ----------------- | ----------------------------------------- |
| **비즈니스 로직** | API 호출, 상태 관리, 검증 등 핵심 로직    |
| **UI 제공**       | 기능을 트리거하는 버튼, 폼 등 UI 컴포넌트 |
| **사이드 이펙트** | 토스트, 네비게이션, 로그 등 액션 후 처리  |
| **상태 관리**     | 해당 기능 전용 상태 및 데이터 관리        |

---

## 2. 구조 · 파일 설명

```
features/
└─ [feature-name]/
   ├─ ui/                # UI 컴포넌트, 데이터 포맷팅
   ├─ api/               # API 호출 로직
   ├─ model/             # 상태 관리, 비즈니스 로직
   ├─ lib/               # 유틸리티 함수
   └─ index.ts           # Public API (필수)
```

| 파일/폴더      | 역할 및 제약사항                                   |
| -------------- | -------------------------------------------------- |
| **폴더명**     | kebab-case 사용 (`login`, `timer-control` 등)      |
| **`ui/`**      | UI 컴포넌트 또는 데이터 포맷팅 함수                |
| **`api/`**     | 외부 API 호출 로직                                 |
| **`model/`**   | 상태 관리, 비즈니스 로직                           |
| **`lib/`**     | 기능 전용 유틸리티 함수                            |
| **`index.ts`** | **Public API 정의. 외부에서는 이것만 import 허용** |

---

## 3. import 규칙

- **허용**: `entities/*`, `shared/*`
- **금지**: `app/*`, `pages/*`, `widgets/*`, `features/*` (다른 feature)

---

## 4. Feature 생성 기준

### 언제 Feature로 만들까?

- **하나의 사용자 액션**을 완결적으로 처리하는 경우
- **재사용 가능한 비즈니스 로직**이 필요한 경우
- 예: 로그인, 로그아웃, 타이머 시작/중지, 파일 업로드

### 언제 다른 곳에 구현할까?

- **페이지 전용 로직**: 해당 페이지의 `ui/` 폴더 내부
- **단순 유틸리티**: `shared/lib/`에 구현 (데이터 포맷팅 등)
- **도메인 로직**: `entities/`에 구현
